# 二、你的微服务项目介绍（面试直接背）
我做过一套**基于 Spring Cloud 微服务架构的电商基础项目**，主要包括：
- 注册配置中心：**Nacos**
- 服务网关：**Spring Cloud Gateway**
- 服务调用：**OpenFeign**
- 统一认证：**JWT + 网关全局鉴权**
- 流量治理：**Sentinel 限流、熔断、降级**
- 异步解耦：**MQ 消息队列（异步、解耦、削峰）**

整体架构：
**前端请求 → 网关统一入口 → 路由转发 → 微服务调用 → 注册中心发现服务**

# 三、每个组件你会怎么说（标准回答）
## 1. Nacos
- 作用：**服务注册发现 + 配置中心**
- 所有微服务启动后注册到 Nacos，网关、Feign 从 Nacos 获取服务地址，实现集群和负载均衡。

## 2. Gateway 网关
- 统一入口、路由转发、**全局鉴权、跨域、限流**
- 所有请求先进网关，网关统一校验 JWT，合法才转发到微服务，避免每个服务都写登录逻辑。

## 3. OpenFeign
- 声明式服务调用，像调用本地方法一样调用远程服务
- 内置 **LoadBalancer 负载均衡**，默认轮询策略
- 配合超时、重试，提升调用稳定性。

## 4. JWT + 网关鉴权
- 登录成功生成 JWT token，前端存在请求头 `Authorization: Bearer xxx`
- 网关全局过滤器统一校验，**无 token、过期、非法直接返回 401**
- 微服务只关心业务，不用处理登录。

## 5. Sentinel（面试必问）
- **限流**：控制接口 QPS，防止流量打垮服务
- **熔断**：下游服务异常、超时、失败率过高，**快速熔断，不调用下游**
- **降级**：熔断/限流后返回友好兜底，不抛错、不雪崩
  一句话：**限流保护自己，熔断防止雪崩，降级提升体验**。

## 6. MQ
- 三大作用：**异步、解耦、削峰**
- 典型场景：订单创建后发 MQ，库存异步扣减，服务之间不强依赖
- 面试常问：消息丢失、重复消费、堆积。

# 四、面试官问：你项目里遇到什么问题？（标准答案）
1. **服务调用雪崩风险**
    - 解决：用 **Sentinel 熔断降级**，异常比例达到阈值自动熔断。

2. **没有统一登录，每个服务都要写鉴权**
    - 解决：**Gateway + JWT 全局统一认证**，网关拦截校验。

3. **同步调用耦合严重，一个挂全挂**
    - 解决：**MQ 异步解耦**，提升系统可用性。

4. **接口被刷、流量过大**
    - 解决：**Sentinel 限流 + 网关层控制**。

# 六、整套微服务流程（面试口述满分）
1. 用户登录，网关放行，生成 JWT 返回前端。
2. 前端请求头带 token，访问网关。
3. 网关过滤器校验 token，合法则路由到对应微服务。
4. 服务通过 OpenFeign 远程调用，Nacos 实现负载均衡。
5. 高并发用 MQ 异步解耦，核心接口用 Sentinel 限流熔断。
6. 整个系统可集群、可扩展、高可用。
